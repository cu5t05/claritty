# Contributing to Claritty

Claritty is intentionally minimal. Its value is in its constraints, not its features. Before contributing, read this document fully.

---

## How Contributions Work

Claritty does not accept unsolicited pull requests. All contributions begin as a conversation, not a code change.

The process:

1. Open a Discussion in the Proposed Contributions category
2. Describe the problem, bug, or security issue clearly
3. If relevant, include a code snippet demonstrating your proposed fix
4. Wait for explicit agreement from the maintainer before writing any code
5. If agreement is reached, you will be directed on how to proceed

Pull requests opened without prior Discussion agreement will be closed automatically.

---

## What Is Accepted

Claritty accepts contributions in two categories only:

- Security patches — vulnerabilities, unsafe behaviors, or execution model gaps
- Bug fixes — behaviors that contradict the documented specification

Feature requests will not be accepted. Claritty's scope is fixed by design. If your proposal expands what the system can do rather than fixing what it already does, it is out of scope.

---

## Rights Declaration

Before any code is incorporated, you must post the following declaration in your Discussion thread:

> "I declare that this code is my original work, contains no third-party or patented material, and I voluntarily relinquish all claims upon its incorporation into Claritty."

This declaration must appear verbatim in the Discussion thread before the maintainer will consider incorporating any code. It is a public, timestamped record of authorship and rights transfer.

---

## No AI-Generated Contributions

Claritty's core principle is that the human decides and the AI proposes — never the reverse. Submitting AI-generated code as a contribution contradicts this principle directly.

AI-generated contributions will not be accepted. By posting a snippet in Discussion, you affirm that the code is your own work and was not generated by an AI tool.

This is not a technical restriction. It is a philosophical one, and it is enforced by the rights declaration above.

---

## Credit

Contributors whose code is incorporated will be credited in CONTRIBUTORS.md and as co-authors in the relevant commit. Credit is given explicitly and deliberately, not automatically.

---

## Forking

If your proposed direction does not align with Claritty's philosophy, you are explicitly encouraged to fork. The license supports it. A fork is not a rejection — it is a legitimate and respected outcome.

The only restriction on forks is the Common Clause: you may not commercialize a fork without a separate agreement with the licensor. Non-commercial forks are unrestricted.
